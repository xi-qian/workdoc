RadixCache

基于基数树的前缀感知 KV 缓存管理系统

❌ 无缓存：每次请求都重新计算

两个用户发送"You are a helpful assistant. "作为系统提示。没有缓存时，每个请求都要从头计算完整的 Attention KV，产生重复计算和冗余显存占用。100 个用户 = 100 倍的系统提示 KV 缓存。

✅ RadixCache：共享公共前缀

RadixCache 识别出两个请求的token 前缀相同，只计算一次 KV 并缓存。第二个请求直接复用已缓存的 KV 索引，跳过 Attention 计算，显存中只存一份系统提示的 KV。

API 核心操作 (radix_cache.py)

match_prefix(key)

radix_cache.py:359

给定一个 token 序列 key，从根节点开始沿树向下遍历，找到 最长公共前缀。返回该前缀对应的所有 KV 缓存页索引的拼接张量。如果匹配结束在某个节点内部，会 split 该节点以暴露精确边界。同时刷新该路径上节点的访问时间戳（供 LRU/LFU 使用）。

insert(key, value, priority)

radix_cache.py:418

将新的 token 序列及其对应的 KV 缓存页索引 value 插入树中。与已有节点共享前缀部分，只创建分叉后的新节点。返回实际新建的前缀长度。新插入的叶子节点被标记为 evictable，加入可驱逐集合。

cache_finished_req(req)

radix_cache.py:438

请求完成后调用。取 req.origin_input_ids + req.output_ids 的完整 token 序列，连同 req_to_token_pool 中的 KV 缓存索引一起插入树中。释放重复的 KV 缓存页（树中已有相同前缀时），并对 req.last_node 执行 dec_lock_ref 解锁。

evict(num_tokens)

radix_cache.py:558

当显存不足需要释放 num_tokens 个位置时调用。从所有 evictable_leaves 中按配置的驱逐策略（LRU/LFU/FIFO/MRU/SLRU/Priority）选择叶子节点，释放其 KV 缓存页 → 删除节点 → 递归清理空父节点。

inc_lock_ref / dec_lock_ref(node)

radix_cache.py:587 / 602

引用计数锁机制：正在被活跃请求使用的节点 lock_ref > 0，不可驱逐。当请求复用某个前缀节点时，对该节点及其所有祖先 inc_lock_ref；请求结束或转移前缀时 dec_lock_ref。lock_ref 降为 0 时节点重新变为可驱逐。

cache_unfinished_req(req)

radix_cache.py:485

用于 chunked prefill 场景：超长序列被切成多块处理，每完成一块就将当前的 KV 缓存插入树中。这允许后续请求立即复用已处理的部分前缀，即使在原请求还未完成时。配合 dec_lock_ref / inc_lock_ref 更新活跃节点引用。

MEM 内存池体系

ReqToTokenPool

mem_cache/memory_pool.py:127

一个 [max_reqs, max_context_len] 的二维张量。
通过 req_pool_idx 索引，每行存储该请求每个 token 位置对应的 KV 缓存页号。请求开始时 alloc()，结束或释放时 free()。

TokenToKVPoolAllocator

mem_cache/allocator.py:121

管理一个空闲页号栈。RadixCache 调用 alloc(num_pages) 获取连续页号，free(indices) 归还。驱逐时释放的页号回到空闲池供后续请求使用。支持 page_size > 1 的对齐分配。

MHATokenToKVPool

mem_cache/memory_pool.py:785

GPU 上的物理 KV 缓存张量。
形状为 [num_pages, page_size, num_heads, head_dim]（MHA）或对应的 MLA 变体。RadixAttention 层通过页号索引直接读写这些张量，完成 KV 缓存的存储和查找。
