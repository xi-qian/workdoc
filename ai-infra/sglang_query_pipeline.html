SGLang Runtime / Query Pipeline

一个 query 进入 SGLang 后的处理流程

本文基于当前仓库源码，追踪从 HTTP/OpenAI 兼容接口收到请求，到 tokenizer、scheduler、model worker、detokenizer，再回到客户端的完整链路。

分析范围：python/sglang/srt 主入口：/generate、/v1/chat/completions 生成日期：2026-05-15
总览 启动结构 HTTP 入口 TokenizerManager Scheduler 模型执行 调度细节 Forward 细节 KV 动画 回传路径 流式与非流式 关键源码

一眼看懂

1. 接收FastAPI 接收 native 或 OpenAI 请求
2. 规范化OpenAI 请求转为内部 GenerateReqInput
3. 调度执行TokenizerManager 经 ZMQ 送入 Scheduler
4. 解码返回Detokenizer 生成文本，唤醒 HTTP 协程返回

简化公式：HTTP 请求 → GenerateReqInput → TokenizedGenerateReqInput → Req → ScheduleBatch → ModelWorker.forward → BatchTokenIDOutput → BatchStrOutput → HTTP Response/SSE。

整体链路图

启动结构

SRT server 启动后，主进程里运行 HTTP server、Engine 和 TokenizerManager；Scheduler 与 DetokenizerManager 是子进程，通过 ZMQ IPC 传对象。

  • Engine 注释明确了三个组件：TokenizerManager、Scheduler、DetokenizerManager。
  • _launch_scheduler_processes 按 TP/PP/DP 拓扑拉起 scheduler 子进程。
  • run_detokenizer_process 拉起 detokenizer 事件循环。

进程职责

HTTP / OpenAI 入口

1

Native /generate

http_server.generate_request 直接接收 GenerateReqInput。如果 stream=true，包装为 StreamingResponse 并持续 yield data: ...；否则只取 generator 的第一个最终结果。

2

OpenAI /v1/chat/completions

FastAPI 路由进入 OpenAIServingChat.handle_request。公共基类先做校验和日志，再调用 _convert_to_internal_request。

3

Chat prompt 转换

OpenAIServingChat 处理 messages、chat template、多模态输入、tool/reasoning 约束，最终构造内部 GenerateReqInput。

4

Completions 转换

OpenAIServingCompletion 处理 prompt、sampling params、logprobs/echo，最终也构造 GenerateReqInput。

TokenizerManager：请求进入引擎前的主控层

TokenizerManager.generate_request 是大部分生成请求的核心入口。它把 HTTP 层对象变成 scheduler 能处理的 tokenized 对象，同时维护异步等待状态。

1

标准化请求

调用 normalize_batch_and_arguments、设置默认优先级、检查 routed_dp_rank。

2

建立请求状态

_init_req_state 为每个 request id 创建 ReqState，后续 detokenizer 回包会写入这个 state 并设置 event。

3

等待 pause / 权重更新锁

如果服务被暂停会等待；进入 model_update_lock.reader_lock 后解析 LoRA，避免与模型热更新冲突。

4

tokenize 与多模态处理

_tokenize_one_request 处理 input_ids、input_embeds、文本 tokenizer、多模态 processor，并产出 TokenizedGenerateReqInput。

5

发送 scheduler

_send_one_request 通过 send_to_scheduler.send_pyobj 把 tokenized 对象推到 scheduler 的 PULL socket。

6

等待响应

_wait_one_response 等待 ReqState.event。流式请求中间 chunk 会直接 yield；非流式请求直到 finished 才返回。

Scheduler：排队、缓存与批处理

Scheduler 子进程从 tokenizer 接收对象，转成内部 Req，进入 waiting queue，再按策略形成 ScheduleBatch 执行。

事件循环

  • event_loop_normal：收请求 → 处理输入 → 取下一批 → 跑 batch → 处理结果。
  • event_loop_overlap：把 CPU 调度与 GPU forward 重叠，降低吞吐瓶颈。
  • recv_requests：入口 rank 从 tokenizer/rpc socket 非阻塞读取，再按 TP/DP/CP 广播。

请求入队

  • TypeBasedDispatcher 把 TokenizedGenerateReqInput 分发到 handle_generate_request。
  • handle_generate_request 创建 Req，补齐 session、bootstrap、multimodal、logprob、grammar 等信息。
  • _add_request_to_queue 根据普通/Prefill/Decode disaggregation 模式放入对应队列。

重要：真正的 batching 不是在 HTTP 层完成，而是在 Scheduler 基于当前 running batch、waiting queue、KV cache、max tokens、chunked prefill、LoRA、优先级等约束动态决定。

模型执行：Prefill 与 Decode

1

选择 batch

get_next_batch_to_run 从 waiting queue 选出可运行请求，创建 ScheduleBatch；同时处理 prefix cache 命中、KV pool 分配、chunked prefill、preemption 等。

2

执行 forward

run_batch 根据 forward mode 调用 model_worker.forward_batch_generation 或 embedding 路径。生成模型通常先 prefill prompt，再 decode 一个或多个 token。

3

采样与结束判断

forward 结果包含 next token、logprobs、finish reason 等。Scheduler 更新请求状态，决定继续 decode、结束、abort、或者流式输出。

4

结果送出

生成路径把 token id 结果封装为 BatchTokenIDOutput 送到 detokenizer；embedding 路径可返回 BatchEmbeddingOutput。

Scheduler 深入：一次调度循环里发生什么

Scheduler 的核心循环可以理解为一个“持续填充 + 持续解码”的状态机。每轮循环先吸收新请求，再优先尝试 prefill；如果没有可 prefill 的新 batch，就推进当前 running decode batch。

1. event_loop_normal / event_loop_overlap

  • event_loop_normal 顺序执行：recv_requests() → process_input_requests() → get_next_batch_to_run() → run_batch() → process_batch_result()。
  • event_loop_overlap 使用 result_queue 让当前 batch 的 GPU forward 与上一 batch 的 CPU 结果处理重叠；遇到连续 prefill、spec+grammar 等情况会关闭重叠。
  • rank 0 从 tokenizer/rpc socket 收包后，会根据 TP/DP/CP 拓扑广播给其他 rank，保证各 rank 看到一致的 work/control request。

2. get_next_batch_to_run 的优先级

阶段关键动作结果
清理处理 waiting/running timeout、过滤 finished/retracted 请求、把上轮 extend batch 合并进 running batch。得到干净的 running_batch。
优先 prefill调用 get_new_batch_prefill，只要有可运行的新请求或 chunked request，就优先构造 EXTEND batch。新 query 更快 TTFT。
否则 decode如果没有 prefill batch 且 running batch 非空，调用 update_running_batch 并准备 DECODE batch。继续为已有请求生成下一个 token。
全局同步DP attention / MLP sync / ngram embedding 等会进一步包装或调整返回 batch。保证分布式 forward 一致。

3. PrefillAdder 如何决定哪些请求能进 batch

  • 优先级：schedule_policy.calc_priority 先对 waiting queue 排序，支持 priority scheduling 与低/高优先级方向配置。
  • 容量：同时受 max_running_requests、pp_max_micro_batch_size、max_prefill_tokens、KV pool 可用 token 数、prefill_max_requests 限制。
  • 缓存：req.init_next_round_input(tree_cache) 计算 prefix cache 命中，prepare_for_extend 只把未命中的后缀 token 放进 prefill。
  • chunked prefill：长 prompt 可被切块，chunked_req 记录未完成的 prefill chunk；动态 chunking 会根据历史长度预测下一块大小。
  • HiCache：启用分层缓存时，会先检查异步 prefetch 是否完成，并记录 device/host/storage 命中明细。
  • LoRA：batch 内 LoRA 数量受 max_loras_per_batch 和 LoRA manager 校验限制，可选择 overlap loading。

4. Decode 内存不足时如何处理

update_running_batch 会先 filter_batch 移除结束请求，然后调用 check_decode_mem 预估下一轮 decode 需要的 KV token。如果不够，会通过 retract_decode 回收部分请求，把它们重新放回队列；极端情况下最后一个请求也放不下，则生成 abort 输出而不是让 scheduler 崩溃。

5. Prefill/Decode 结果处理差异

处理函数输入核心动作
process_batch_result_prefillEXTEND batch 的 logits 和 next token把首个生成 token 写入 req.output_ids；处理 input logprobs、hidden states、grammar；未结束请求写入 prefix cache 并进入后续 decode。

模型执行深入：从 ScheduleBatch 到 logits / next token

Scheduler 并不直接调用具体模型文件，而是先把 ScheduleBatch 转成 ModelWorkerBatch，再由 TpModelWorker 构造 ForwardBatch，最后进入 ModelRunner.forward。

1

ScheduleBatch.get_model_worker_batch

把 scheduler 层的请求、KV pool 索引、seq_lens、sampling info、LoRA、多模态、spec 信息打包成 worker 可用的 batch。

2

TpModelWorker.forward_batch_generation

调用 ForwardBatch.init_new 生成真正传入模型的张量视图；如果是 PP 最后一 rank，则负责得到 logits 并采样 next token，否则只向后传 pipeline proxy tensors。

3

ModelRunner.forward

包裹 profiler、expert distribution recorder、routed experts/indexer capture、elastic EP 恢复等横切逻辑，然后调用 _forward_raw。

4

_forward_raw

优先检查 decode CUDA graph 能否 replay；否则准备 MLP sync 或 attention TP scatter，初始化 attention metadata，再按 forward mode 走 decode/extend/split-prefill/idle。

5

sample

对 logits 应用 grammar vocab mask、logits bias、penalty/sampling 参数，然后由 sampler 产出 next_token_ids，并写回 logprob 相关字段。

Forward mode 对照

Forward mode何时使用模型输入主要优化
EXTEND新请求 prefill、长 prompt chunk、prefill-only 请求未缓存的 prompt 后缀 token，positions 是连续区间prefix cache 跳过已算 token；piecewise CUDA graph；chunked prefill。
DECODErunning batch 每轮生成下一个 token上一轮输出 token，每个请求通常 1 token固定 batch/shape 时 replay CUDA graph；每轮只分配一个 KV slot。
MIXEDmixed chunked prefill 开启且可与 running decode 合批prefill token + decode token 拼接提高 GPU 利用率，但避开 logprob/input_embeds 等不兼容场景。
SPLIT_PREFILLPD multiplexing 或特殊 split prefill按 split 状态分段传递 hidden/residual把大 prefill 拆成更可控的 forward 片段。

prepare_for_extend：prefill 张量如何准备

  • 计算每个请求的 prefix_lens 与 extend_lens，只把 fill_ids[len(prefix_indices):] 放进 input_ids。
  • 调用 alloc_for_extend 分配 request pool slot 与 KV cache 输出位置，并更新 req 级别 kv_committed_len / kv_allocated_len。
  • 处理 input_embeds、position embedding override、多模态 MultimodalInputs、encoder-decoder 信息、Mamba tracking。
  • 如果需要 logprob，会构造 extend_input_logprob_token_ids，用于计算输入 token 的 logprob。

prepare_for_decode：decode 张量如何准备

  • 把 forward_mode 设置为 DECODE，清理 prefill 阶段残留的 input_embeds 与 context parallel metadata。
  • 对 penalties/orchestrator 累计输出 token，然后把上一轮 output_ids 作为本轮 input_ids。
  • 调用 alloc_for_decode(token_per_req=1) 为每个请求追加一个 KV cache 位置。
  • 更新 seq_lens、orig_seq_lens、seq_lens_sum，overlap 模式避免 in-place 更新以减少并发风险。

分支逻辑输出

图例
蓝色：prompt token，本轮可能被整体 prefill。
绿色：历史 K/V 被直接复用，没有重算。
金色：当前轮新写入的 K/V 槽位。
青色：decode 新生成 token，会在下一轮成为历史上下文。

KV cache 的本质是“把每个历史 token 的 K/V 张量存起来”。下一轮 decode 不再重新前向整个前缀，而是只为最新 token 计算一次 Q/K/V，然后用新 Q 去和缓存里的所有历史 K/V 做 attention。

因此 decode 的复杂度没有降到常数。单轮仍要读取越来越长的历史 K/V，但避免了“每轮重算整个前缀”的巨大重复计算，这就是大模型自回归推理必须依赖 KV cache 的原因。

把动画映射回 SGLang 代码

  • prepare_for_extend 会根据 prefix_indices 只把未命中的后缀 token 送入 prefill，同时分配 out_cache_loc，把新算出的 K/V 写进 KV pool。
  • prepare_for_decode 每轮只把上一轮输出 token 作为新的 input_ids，再通过 alloc_for_decode(token_per_req=1) 追加一个 cache 槽位。
  • ForwardBatch 持有 req_pool_indices、seq_lens、out_cache_loc 等信息，attention backend 正是靠这些索引在 KV pool 中定位历史 token。
  • check_decode_mem / retract_decode 说明 KV cache 是一个有容量上限的池；放不下新 token 时，scheduler 必须回收或撤回部分请求。

Radix Cache 关键点
命中的前缀 token 不需要再次 prefill。

在 SGLang 里，radix cache 不是替代 KV cache，而是建立在它之上的“跨请求前缀复用层”。命中前缀后，Scheduler 会把已缓存前缀映射成 prefix_indices，prepare_for_extend 只把后缀 token 放进 input_ids。

命中越长，TTFT 越低，prefill FLOPs 越少。但一旦请求分叉到不同后缀，分叉之后的 token 仍要正常 prefill，并在 KV pool 中分配新的 cache 位置。

回传路径：Token IDs 到 HTTP 响应

1

Detokenizer 接收

DetokenizerManager.event_loop 从 scheduler 接收 BatchTokenIDOutput，通过 dispatcher 调用 handle_batch_token_id_out。

2

增量 decode

_decode_batch_token_id_output 维护每个 rid 的 DecodeStatus，处理 UTF-8 边界、stop trim、batch decode，并得到 text delta。

3

返回 TokenizerManager

Detokenizer 构造 BatchStrOutput，通过 tokenizer IPC socket 发回主进程。

4

唤醒等待协程

TokenizerManager.handle_loop 收到 BatchStrOutput 后调用 _handle_batch_output，更新 ReqState.out_list，设置 state.event。

5

HTTP 层 yield

_wait_one_response 被唤醒后 yield chunk 或最终结果；FastAPI 将其编码为 JSON 响应或 SSE 数据帧。

流式请求

  • Native /generate 返回 StreamingResponse，每次输出 data: {json}\n\n，最后输出 data: [DONE]。
  • OpenAI chat/completions 会把内部 chunk 重新包装为 OpenAI SSE chunk。
  • 若开启 incremental_streaming_output，text 和 output_ids 是 delta；否则需要按 offset 切出新增文本。
  • 断连时后台 abort task 或 request.is_disconnected() 会触发 abort。

非流式请求

  • HTTP handler 只取 generate_request(...).__anext__()。
  • _wait_one_response 在非流式模式下忽略中间 chunk，直到 finished=true。
  • 最终响应包含完整 text、output_ids 与 meta_info。
  • OpenAI wrapper 再把内部结果转成 ChatCompletionResponse 或 CompletionResponse。

异常、取消与控制点

  • 请求断连：_wait_one_response 周期性检查 request.is_disconnected()，必要时调用 abort。
  • 流式取消：StreamingResponse 挂载 create_abort_task，客户端断开时通知 scheduler。
  • 输入过长：Scheduler validate_input_length 失败时把请求标记为 abort，再走正常输出链路返回错误。
  • 服务暂停：TokenizerManager 在 is_pause_cond 上等待，暂停期间不继续向 scheduler 派发新请求。
  • 模型更新：model_update_lock.reader_lock 保护生成请求与权重/LoRA 更新互斥。
