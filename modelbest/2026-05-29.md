# Qwen3.6-35B-A3B + Claude Code 集成工作总结

**日期：** 2026-05-29
**环境：** Jetson AGX Orin (64GB, CUDA 12.6) → 本地 Claude Code

---

## 一、背景与目标

将本地 Jetson AGX Orin 上的 Qwen3.6-35B-A3B 模型接口改造为 Claude Code 兼容，替代原有 OpenAI 兼容端点，直接使用 Anthropic Messages API。

---

## 二、工作内容

### 1. 验证 llama.cpp 原生 Anthropic 支持

**结论：llama.cpp v9374 已内置 Anthropic Messages API，无需额外 proxy。**

通过 curl 验证了以下功能：

| 功能 | 状态 | 说明 |
|------|------|------|
| 纯文本对话 | ✅ | `/v1/messages` 正常返回 |
| 流式 SSE | ✅ | 完整支持 `message_start`、`content_block_delta`、`message_stop` 等事件 |
| Tool Calling | ✅ | 通过 chat template 软约束实现 |
| Thinking | ✅ | `reasoning_content` 自动映射为 `thinking` 块 |
| Vision | ✅ | 支持 base64 图片输入 (需 mmproj) |
| Token Counting | ✅ | `/v1/messages/count_tokens` 端点可用 |
| Prompt Caching | ⚠️ | 接受 `cache_control` header 但**无实际缓存** |
| Beta Features | ❌ | 不支持 |

### 2. Claude Code 项目配置

**配置文件：** `.claude/settings.local.json`

| 配置项 | 值 | 目的 |
|--------|-----|------|
| `ANTHROPIC_BASE_URL` | `http://192.168.3.241:8080` | 指向 Jetson 上的 llama-server |
| `ANTHROPIC_AUTH_TOKEN` | `"none"` | 避免与 shell 环境变量冲突 |
| `ANTHROPIC_MODEL` | `"qwen"` | 指定使用的模型 |
| `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC` | `1` | 禁止向 Anthropic 服务器发送遥测 |
| `CLAUDE_CODE_DISABLE_EXPERIMENTAL_BETAS` | `1` | 去掉 llama-server 不支持的 beta header |
| `DISABLE_PROMPT_CACHING` | `1` | llama-server 无 prompt caching，避免客户端尝试 |
| `API_TIMEOUT_MS` | `3000000` | 50 分钟超时（本地推理约 30 tokens/s） |
| `DISABLE_TELEMETRY` | `1` | 关闭遥测 |
| `DISABLE_GROWTHBOOK` | `1` | 关闭 feature flag 拉取 |

**认证冲突修复：** Shell 环境已设置 `ANTHROPIC_AUTH_TOKEN`（指向 `https://open.bigmodel.cn/api/anthropic`），项目配置使用 `ANTHROPIC_AUTH_TOKEN: "none"` 覆盖，避免同时存在 token 和 API key 导致告警。

### 3. Prompt Caching 分析

**llama.cpp 对 Anthropic API 各特性的支持情况：**

| 特性 | llama.cpp 支持度 | 说明 |
|------|-----------------|------|
| Streaming SSE | ✅ 完整支持 | |
| Tool use | ✅ 软约束 (chat template) | 复杂场景可能出错 |
| Thinking | ✅ | reasoning_content 自动映射 |
| Vision | ✅ | 需 mmproj |
| **Prompt Caching** | ❌ 无效 | 接受 `cache_control` 但 `cache_read_input_tokens` 仅在有缓存时有效，无实际缓存机制 |
| Beta Features | ❌ | |

**Proxy 无法解决的问题：** Prompt Caching 的核心是服务端维护 token 哈希缓存，属于服务端能力而非协议转换。Proxy 只能转发请求/响应，无法为 llama.cpp 补充缓存机制。

**替代方案对比：**

| 特性 | llama.cpp (当前) | vLLM | SGLang |
|------|-----------------|------|--------|
| Prompt Caching | ❌ | ✅ | ✅ |
| Tool Calling | ⚠️ 软约束 | ✅ 硬约束 (Grammar) | ✅ 硬约束 (Grammar) |
| Thinking | ✅ | ✅ | ✅ |
| Anthropic API | ✅ 原生 | ✅ 原生 | ✅ 原生 |
| Jetson 成熟度 | ✅ 最成熟 | ✅ 较好 | ⚠️ 实验性 |

### 4. 服务管理

**问题排查：**

- 发现 `llama-cli` 进程（pid 24964）占满 28GB 内存 + 99.5% CPU，挤掉了 `llama-server`
- 该进程是之前用户手动执行的测试命令：`llama-cli -m <model> -p test -n 1`
- 杀掉后重启服务，确认端口 8080 可用

**配置 systemd 用户态服务：**

创建 `~/.config/systemd/user/llama-server.service`：

```ini
[Unit]
Description=llama.cpp Server (Qwen3.6-35B-A3B)
After=network.target

[Service]
Type=simple
WorkingDirectory=/data/llama.cpp
Environment=LD_LIBRARY_PATH=/data/llama.cpp/build/bin
ExecStart=/data/llama.cpp/build/bin/llama-server \
  -m /data/models/Qwen3.6-35B-A3B-GGUF/Qwen3.6-35B-A3B-UD-Q4_K_M.gguf \
  --mmproj /data/models/Qwen3.6-35B-A3B-GGUF/mmproj-F16.gguf \
  --host 0.0.0.0 --port 8080 \
  -ngl 99 -c 131072 \
  --reasoning-format deepseek \
  --parallel 1 --timeout 600
Restart=on-failure
RestartSec=10
StandardOutput=append:/data/llama-server.log
StandardError=append:/data/llama-server.log

[Install]
WantedBy=default.target
```

**关键配置：**
- `Restart=on-failure` + `RestartSec=10` — 崩溃后 10 秒自动重启
- `loginctl enable-linger modelbest` — 用户不登录时服务也在后台运行
- 服务状态：`active (running)`，占用 20.3GB 内存

### 5. 上下文长度与内存测试

**实际配置：** `-c 131072` (128K)，与模型训练上下文 256K 一致。

**测试方法：** 发送 ~280K 字符（约 120K tokens）的超长请求，同时监控内存占用。

**测试结果：**

| 状态 | 总内存占用 | 可用内存 | 说明 |
|------|-----------|---------|------|
| 服务空载 | 28GB | 31GB | 模型 ~21GB + KV cache ~7GB |
| 发送 120K tokens 后 | 28GB | 32GB | KV cache 预分配，不随输入增长 |
| 缓存命中（第二次） | 28GB | 32GB | 处理时间从 120s 降至 1.5s |

**结论：** KV cache 在启动时预分配，实际输入长度变化不影响内存占用。128K 配置在 64GB Jetson 上充裕。

**Anthropic 端点 token 计数验证：**

| 场景 | cache_read_input_tokens | input_tokens | 总计 | 说明 |
|------|------------------------|-------------|------|------|
| 首次（缓存清空） | 0 | 120010 | 120010 | 正确 |
| 缓存命中 | 120006 | 4 | 120010 | 正确 |

Anthropic 端点的 token 计数使用 Anthropic 语义：`input_tokens` = 新处理 token，`cache_read_input_tokens` = 缓存命中 token，两者之和 = 总 token 数。计数准确。

## 三、已知限制

1. **Tool Calling 稳定性** — llama.cpp 使用 chat template（软约束），非 grammar 硬约束。复杂多工具场景可能输出非法 JSON。如需 100% 准确，需迁移到 vLLM/SGLang。
2. **Prompt Caching 无效** — llama.cpp 无 token 缓存机制，相同 prompt 每次需重新计算。
3. **单并发** — 64GB 内存限制，多请求需排队。
4. **首次推理慢** — 首次 prompt 处理约 10-20 秒。
5. **Thinking 为英文** — Q4_K_M 量化后思考过程输出英文，最终回答中文。

---

## 四、后续建议

如需要解决 tool calling 稳定性和 prompt caching 两个核心问题，建议评估迁移到 SGLang（Jetson 支持前缀树缓存，多轮场景更省电）或 vLLM（更成熟稳定）。
