NanoClaw Message Processing Flow

Inbound (top) → Container Processing (middle) → Outbound (bottom). Two-DB session split is the only IO surface between host and container.

① Inbound — Routing

  • • Adapter normalizes platform event → InboundEvent
  • • Router: messaging-group lookup → sender resolve → agent fan-out
  • • Engage eval per wiring: pattern · mention · mention-sticky
  • • Permission hooks: access gate + sender scope gate
  • • Command gate: filter / admin-only / pass classification
  • • Session resolves (shared / per-thread / agent-shared)
  • • Write to inbound.db (even seq) → wakeContainer()

② Container — Agent Processing

  • • Bun runtime, per-session DB bind-mounts (DELETE journal)
  • • Poll loop: read pending → ack processing → format XML
  • • Provider (claude-agent-sdk) MessageStream + tool hooks
  • • MCP tools: core, agents, cli, interactive, scheduling, self-mod
  • • Concurrent polling pushes follow-up messages mid-query
  • • Output parsed to <message to="..."> → writeMessageOut
  • • Heartbeat = file touch (cross-mount safe, not DB)

③ Outbound — Delivery + Sweep

  • • Two poll loops: active 1s (running) · sweep 60s (all)
  • • System actions: scheduling, approvals, agent route, self-mod
  • • Channel adapter: text, cards, reactions, edits, files
  • • markDelivered() in inbound.db → idempotent
  • • Host sweep: ack sync, due wake, stuck kill, crash reset, recurrence
  • • Response registry: approval / sender-approval / ask-question replies
  • • Replies route back via router interceptor → claim by handler
