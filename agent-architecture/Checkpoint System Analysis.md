# Checkpoint 系统分析

## Intuition

Checkpoint 是 Hermes Agent 的文件系统安全网——在智能体修改或删除用户文件之前，自动对工作目录拍快照，让用户可以通过 `/rollback` 回滚。类似于编辑器的"自动保存历史"，只不过保存的是整个项目目录的 git 快照，而不是单个文件。

## 整体设计

### 存储架构

所有快照数据存放在 `~/.hermes/checkpoints/store/` 这个共享影子 git 仓库中，不碰用户项目的 `.git/`。

```
~/.hermes/checkpoints/store/          # 共享裸仓库（bare repo）
├── HEAD
├── objects/                           # git 对象（跨项目去重）
├── refs/hermes/<hash16>              # 每个项目的快照提交链
├── indexes/<hash16>                  # 每个项目独立的 git 索引
└── projects/<hash16>.json            # 项目元数据（workdir, created_at, last_touch）
```

### 项目识别

"项目"通过启发式标记文件向上查找确定：

```python
markers = {".git", "pyproject.toml", "package.json", "Cargo.toml",
           "go.mod", "Makefile", "pom.xml", ".hg", "Gemfile"}
```

从被修改文件的父目录开始，逐级向上找到第一个包含标记文件的目录，作为项目根。找不到就以文件所在目录兜底。

项目根路径经 SHA256 取前 16 位后作为唯一标识，映射到影子仓库中的独立 ref、索引和元数据。

### 触发时机

快照在以下操作执行**之前**触发：

| 触发条件 | 检测方式 |
|---------|---------|
| `write_file` / `patch` 工具调用 | 直接拦截 |
| 破坏性终端命令 | 正则匹配（`rm`, `mv`, `sed -i`, `>` 重定向等） |

去重规则：**每个对话轮次、每个项目目录最多拍一次**。同一轮次内，无论对同一项目下多少个文件执行写操作，只拍第一张快照。

### 容量控制

| 参数 | 默认值 | 作用 |
|------|-------|------|
| `max_snapshots` | 20 | 单项目最大快照数，超出淘汰最旧的 |
| `max_total_size_mb` | 500 | 全局存储上限，超出按轮询淘汰各项目最旧快照 |
| `max_file_size_mb` | 10 | 单文件超过此大小不纳入快照 |
| `_MAX_FILES` | 50000 | 目录文件数超过此值跳过整个目录 |

自动维护：启动时执行 `maybe_auto_prune_checkpoints()`，每 24 小时最多一次，清理孤立项目和过期快照。

### 用户交互

| 操作 | 方式 |
|------|------|
| 列出快照 | `/rollback` |
| 回滚到某快照 | `/rollback <N>` |
| 查看 diff | `/rollback diff <N>` |
| 回滚单个文件 | `/rollback <N> <file>` |
| 命令行清理 | `hermes checkpoints prune/clear/clear-legacy` |

回滚前会自动再拍一张快照（pre-rollback snapshot），支持"撤销撤销"。

## 版本演进

| 版本 | 时间 | 设计 | 问题 |
|------|------|------|------|
| v1 | 2026-03-10 | 每个项目一个独立影子仓库 | 零去重，同内容跨项目重复存储；pruner 是空操作，对象只增不减 |
| v2 | 2026-05-06 | 单一共享仓库 + 独立 ref/index | 跨项目 git 对象去重；pruner 真正生效；全局容量上限 |

## 设计评价

### 做得好的部分

1. **影子仓库隔离**：通过 `GIT_DIR` / `GIT_WORK_TREE` / `GIT_INDEX_FILE` 环境变量重定向，完全不碰用户的 `.git/`，不影响用户项目的 git 状态。这个设计很干净。

2. **v2 的去重改进**：单一共享仓库让跨项目的相同文件只存一份 git 对象，解决了 v1 的空间浪费问题。

3. **去重策略合理**：每轮次每项目只拍一次，避免了对同一轮次内多次小修改产生大量冗余快照。

4. **回滚安全**：回滚前自动拍快照，防止用户回滚后后悔。

### 核心问题

#### 1. 静默跳过，用户无法感知

这是最大的问题。快照在多种条件下会被跳过，但所有跳过都走 `logger.debug()`，用户完全无感知：

- 系统没有 git → 跳过
- 目录是 `/` 或 `~` → 跳过
- 目录文件数 > 50000 → 跳过
- 单文件 > 10MB → 不纳入
- 拍完发现没有变化 → 跳过
- 拍摄过程出错 → 跳过

用户开了 checkpoints 功能，自然预期每次写操作前都有快照。但实际上各种条件会无声跳过，等真要 `/rollback` 时才发现没有快照——"保险"在需要的时候不在。

#### 2. 项目识别是启发式的，无法覆盖所有情况

标记文件列表是硬编码的，覆盖了主流项目类型（Python、Node、Rust、Go、Java、Ruby），但：
- 没有标记文件的项目（纯 C 项目、自定义构建系统）会兜底到文件所在目录
- 同一目录下多个标记文件不会误判（取最内层），但嵌套项目的边界可能不符合用户预期（比如在 monorepo 的子目录里工作）

#### 3. 去重粒度可能丢失中间状态

"每轮次每项目只拍一次"意味着：智能体在同一轮次内先修改 A 文件，再修改 B 文件，只有修改 A 之前的快照。如果 A 的修改破坏了东西，用户可以回滚；但如果 B 的修改破坏了东西，回滚会把 A 的修改一起撤销。快照粒度不够细，但这是性能的权衡。

#### 4. 破坏性命令检测靠正则匹配

终端命令是否触发快照依赖 `_is_destructive_command()` 的正则匹配。这种匹配天然不完整：
- 漏判：`python -c "import shutil; shutil.rmtree('dir')"` 不会被识别为破坏性
- 误判：`echo "rm something"` 只是打印字符串，但可能被匹配

正则匹配无法覆盖所有破坏性操作，也无法排除所有非破坏性操作。

#### 5. 默认关闭

Checkpoint 默认不启用，大多数用户可能根本不知道有这个功能。

## 改进建议

### 短期：让跳过可感知

跳过时给用户一个提示，而不是静默吞掉。比如：

```
⚠ Checkpoint skipped for /path: directory has >50000 files. Consider adding .gitignore.
```

或者在 `/rollback` 列表中标注哪些轮次没有快照及原因。这样用户至少知道哪些操作是"没保险"的。

### 中期：补全触发覆盖面

- 对 `execute_code` 等工具也加入 checkpoint 触发，而不仅是 `write_file` / `patch` / 终端
- 考虑让用户配置哪些工具需要触发快照（白名单机制）

### 长期：更精细的快照粒度

允许用户选择快照策略：
- **每轮次一次**（当前行为，性能优先）
- **每次写操作一次**（安全优先，快照更多）
- **手动触发**（用户自己决定什么时候拍）

这样用户可以根据项目大小和风险容忍度自行选择，而不是一刀切。
