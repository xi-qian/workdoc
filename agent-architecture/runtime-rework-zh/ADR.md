# 运行时重构架构决策记录

本文件记录必须在上下文丢失后仍然存续的决策。未来的实现不应在没有显式新架构决策记录条目的情况下推翻这些决策。

## ADR-001: 先做租户与技能边界，再做运行时隔离

决策：在实现新运行时之前，先将平台代码与租户/智能体服务/业务技能分离。

原因：

- 租户是一个部署、配置和技能管理层。
- 运行时挂载和权限策略取决于路径是平台代码、租户管理的配置、智能体服务配置、分组运行时状态，还是密钥状态。
- 先做运行时会导致权限规则变成猜测，引发返工。

状态：已采纳。

## ADR-002: 在新运行时达到功能对等之前，保持分组 Docker 运行时

决策：引入 `RuntimeDriver`，并将 `docker-per-group` 保持为默认方案，直到 Version 2.0。

原因：

- 提供回滚能力。
- 允许每个版本独立发布。
- 防止运行时拓扑变更掩盖配置/IPC 缺陷。

状态：已采纳。

## ADR-003: 使用 DB-backed IPC 作为新的核心 IPC

决策：将核心宿主机-智能体/分组消息交换迁移到 SQLite 的 `inbound.db`、`outbound.db` 和 `state.db`。

原因：

- 文件队列和哨兵文件在热进程和隔离任务场景下不够健壮。
- SQLite 提供持久化状态、重试、排序和审计追踪。
- NanoClaw 2.0 已验证此方向。

状态：已采纳。

## ADR-004: 临时保留旧文件 IPC

决策：在迁移业务工具期间，保留旧文件 IPC 作为兼容层。

原因：

- 现有飞书/业务技能依赖文件请求/结果目录。
- 一次性迁移所有工具风险过高。
- DB 核心可以先发布，同时工具桥接迁移逐步推进。

状态：已采纳。

## ADR-005: 每个智能体服务一个 Docker 容器，分组使用独立 Linux 用户

决策：最终运行时每个智能体服务使用一个 Docker 容器/服务。在智能体容器内部，分组通过各自的 Linux 用户进行隔离。

原因：

- Docker 仍然是每个智能体服务的宿主机隔离边界。
- 多个智能体应为多个 Docker 服务。
- 分组用户比分组 Docker 容器更轻量。
- 简单用户工作负载受益于更低的热启动延迟和更低的空闲开销。

权衡：

- 隔离强度低于每个分组一个容器。
- 默认无分组级别的网络或挂载命名空间。

状态：已采纳。

## ADR-006: 隔离任务使用与分组相同的用户

决策：隔离任务以与其父分组相同的 Linux 用户身份运行，但拥有独立的运行时目录、DB、续接和生命周期。

原因：

- 当前语义意味着全新上下文，而非更强的安全性。
- 任务应能访问分组可访问的相同文件和工具。
- 为每个任务创建单独用户会使所有权管理复杂化，且不符合产品语义。

状态：已采纳。

## ADR-007: 隔离任务不得复用实时续接

决策：`context_mode: "isolated"` 不得读取或写入实时聊天续接。

原因：

- 当前行为是全新会话/无聊天历史。
- 任务提示必须自包含。
- 防止定时/后台工作污染实时对话上下文。

状态：已采纳。

## ADR-008: 监督进程拥有分组进程生命周期

决策：在智能体容器运行时中，宿主机代码不直接启动分组进程，而是向相关智能体容器中的监督进程发起请求。

原因：

- 用户创建、chown、chmod、进程启动、停止和状态查询是容器内部职责。
- 宿主机不应需要了解容器内部用户 ID 或进程树细节。
- 实现一个稳定的 RuntimeDriver API。

状态：已采纳。

## ADR-009: Provider 差异封装在 AgentProvider 之后

决策：Claude、OpenCode 及未来的 Provider 必须实现 `AgentProvider`；调度器/路由器不应包含 Provider 特定的分支逻辑。

原因：

- Claude Agent SDK 和 OpenCode 有不同的会话、钩子、工具和事件模型。
- 将 Provider 差异隔离可防止运行时代码变为 Provider 特定。

状态：已采纳。

## ADR-010: OpenCode 在成为默认选项之前是可选的

决策：先以可选方式添加 OpenCode Provider。Claude 保持可用。

原因：

- OpenCode 降低了简单工作负载的开销，但不是 Claude Agent SDK 的即插即用替代品。
- 钩子、恢复和子智能体行为存在差异。
- 生产部署需要回退方案。

状态：已采纳。

## ADR-011: 密钥保留在宿主机/监督进程侧

决策：租户仓库可以引用密钥，但不能存储密钥值。分组可读文件不应包含共享密钥。

原因：

- 业务技能仓库可能被共享或版本管理。
- 分组用户不应读取其他分组或智能体服务的密钥。
- 凭证代理或作用域环境变量注入更安全。

状态：已采纳。

## ADR-012: 用户隔离后不再使用全局可写运行时 IPC

决策：在 Version 1.6 之后，分组运行时目录不得依赖 `0777` 目录或 `0666` 文件。

原因：

- 全局可写 IPC 破坏 Linux 用户隔离。
- 分组所有权加监督进程分组访问是智能体容器内预期的安全边界。

状态：已采纳。

## ADR-013: 注册分组不等于活跃运行

决策：容量规划和运行时状态必须区分已配置分组与活跃运行。

原因：

- 已停止的空闲分组不应消耗 RAM。
- 新架构优化活跃的轻量工作负载。
- 调度和监控需要队列/运行状态，而非仅分组注册状态。

状态：已采纳。

## ADR-014: 版本文档是阶段范围的权威依据

决策：每个 `0x-*.md` 文档定义该阶段的目标和非目标。实现不应提前拉取后续阶段的工作，除非完成当前阶段所必需。

原因：

- 防止大范围重写。
- 保持回滚容易。
- 使进展可审查。

状态：已采纳。

## ADR-015: 租户技能以只读智能体服务输入进入运行时

决策：租户管理和智能体管理的技能在部署/配置加载时解析，并以只读方式挂载到智能体服务容器中。分组用户可以读取这些技能，但不能修改。

原因：

- 租户是管理层，不是运行时隔离单元。
- 一个智能体服务容器可以服务共享该智能体配置技能集的多个分组。
- 分组用户对租户技能仓库的写权限将绕过审查，使配置漂移难以审计。

状态：已采纳。

## ADR-016: 生成的技能在提升之前是分组局部的

决策：运行时由分组创建或修改的技能存储在该分组的运行时目录下，不会自动复制到租户仓库。

原因：

- 运行时生成的内容不应变更部署的真实来源。
- 提升为租户技能应显式且可审查。
- 分组局部的生成技能保持隔离预期。

状态：已采纳。

## ADR-017: 中央宿主机 DB 在迁移期间保持为控制面真实来源

决策：每次运行的运行时 DB 是队列和运行局部状态。现有宿主机 DB 在后续显式迁移替换之前，仍然是聊天、消息历史、定时任务、任务运行日志、注册分组、路由器游标和旧会话 ID 的权威来源。

原因：

- 宿主机消息循环当前使用 `last_timestamp` 和 `last_agent_timestamp` 来避免重复回复并重试失败运行。
- 定时任务、注册分组、发送方策略、卡片操作和通道元数据是宿主机级控制面数据，而非单次运行时数据。
- 将运行时 DB 视为第二个真实来源会导致游标漂移和重复投递风险。

状态：已采纳。

## ADR-018: 宿主机侧授权检查保留在宿主机侧

决策：发送方白名单、触发规则、主分组权限、审批白名单、挂载白名单、分组注册权限和通道凭证检查仍由宿主机或监督进程在执行工作之前执行。

原因：

- 分组进程不受信任自行报告授权状态。
- 飞书和通道凭证是宿主机侧能力。
- 主/自身授权语义已存在于文件 IPC 桥中，必须在工具 IPC 迁移中保留。

状态：已采纳。

## ADR-019: 运行时代码默认不可变

决策：在最终的智能体容器运行时中，监督进程、Provider、MCP 和智能体运行器平台代码由镜像拥有且为只读。分组创建的行为应移至分组生成的技能或分组记忆，而非可写的运行器源码。

原因：

- NanoClaw 1.0 将 `container/agent-runner/src` 复制到每个分组的可写目录中。将这种做法带入共享智能体容器会让一个分组以难以审计的方式变更运行时代码。
- 技能提供更小且可审查的自定义边界。
- 保持平台代码不可变使 Provider 和监督进程升级可预测。

状态：已采纳。

## ADR-020: 额外挂载范围必须显式声明

决策：额外宿主机挂载必须声明是智能体级还是分组级。分组级挂载不得在单容器智能体运行时中静默提升为智能体级挂载。

原因：

- 在 NanoClaw 1.0 中，挂载实际是分组级的，因为每个分组有各自的 Docker 容器。
- 在新运行时中，一个智能体容器可以服务多个分组，因此智能体级挂载对每个分组用户可见，除非额外权限加以阻止。
- 现有的外部挂载白名单和非主只读策略是安全模型的一部分。

状态：已采纳。
